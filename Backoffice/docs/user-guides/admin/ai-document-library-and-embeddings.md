# AI Document Library and Embeddings

This document describes how the AI Document Library works: how documents are processed, how embeddings (vectors) are generated, and how they are used for semantic search and RAG (Retrieval-Augmented Generation).

## Overview

The AI Document Library lets admins upload documents (PDF, Word, etc.) so the chatbot and AI tools can answer questions using that content. Documents are:

1. **Extracted** — text, pages, sections, and optionally tables are extracted from the file.
2. **Chunked** — split into smaller text chunks for context and token limits.
3. **Embedded** — each chunk is turned into a vector (embedding) by an embedding model.
4. **Stored** — vectors are stored in the database (pgvector) and linked to chunks.

When a user asks a question, the query is embedded with the same model, and the system finds the most similar chunks (vector or hybrid search) and uses them as context for the answer.

## Document processing pipeline

Processing runs when you:

- **Upload** a document on the AI Document Library page (`/admin/ai/documents`).
- **Process selected** documents imported from the Document Management system.
- **Reprocess** an existing document (re-extract, re-chunk, re-embed).

Pipeline steps (see `app/routes/ai_documents.py`, `_process_document_sync`):

| Step | What happens |
|------|----------------|
| **1. Extract** | `AIDocumentProcessor` reads the file and extracts text, page boundaries, sections, and (optionally) tables. |
| **2. Chunk** | `AIChunkingService` splits the text into chunks (default: semantic chunking, ~512 tokens per chunk, 50-token overlap). Optional: table chunks and UPR visual chunks. |
| **3. Generate embeddings** | Only if the document is marked *searchable*. `AIEmbeddingService` turns each chunk's text into a vector (see [Embedding providers](#embedding-providers) below). |
| **4. Store** | `AIVectorStore` saves each vector in the `AIEmbedding` table (pgvector), linked to the corresponding `AIDocumentChunk`. |

Chunk records (`AIDocumentChunk`) are always created; embeddings are only created when the document is searchable.

## Embedding providers

Embeddings are generated by `app/services/ai_embedding_service.py`. Two providers are supported.

### OpenAI (default)

- **Provider:** `AI_EMBEDDING_PROVIDER=openai` (default).
- **Model:** `AI_EMBEDDING_MODEL` — default `text-embedding-3-small`.
- **Dimensions:** `AI_EMBEDDING_DIMENSIONS` — default `1536` (must match the pgvector column).
- **Where it runs:** On **OpenAI's servers**. Your app sends chunk text to the OpenAI API and receives vectors. You need `OPENAI_API_KEY` set.
- **Cost:** Billed per token by OpenAI (e.g. text-embedding-3-small is low cost per 1M tokens).

So **`text-embedding-3-small` is not run locally** — it is used via the OpenAI API.

### Local (Sentence Transformers)

- **Provider:** `AI_EMBEDDING_PROVIDER=local`.
- **Model:** `all-MiniLM-L6-v2` (fixed in the service) — 384 dimensions.
- **Where it runs:** **On your machine**. The app loads the model with the `sentence_transformers` library; no external API is called.
- **Cost:** No per-token cost; requires `pip install sentence-transformers` and enough RAM/CPU.

If you use local embeddings you must:

- Set `AI_EMBEDDING_DIMENSIONS=384`.
- Ensure the database pgvector column is 384 dimensions (migration may be required). The default schema is 1536 for OpenAI.

## Configuration reference

| Variable | Default | Description |
|----------|---------|-------------|
| `AI_EMBEDDING_PROVIDER` | `openai` | `openai` or `local`. |
| `AI_EMBEDDING_MODEL` | `text-embedding-3-small` | Model name (OpenAI only; ignored for `local`). |
| `AI_EMBEDDING_DIMENSIONS` | `1536` | Vector size. Must match the DB column and the model (1536 for text-embedding-3-small, 384 for all-MiniLM-L6-v2). |
| `OPENAI_API_KEY` | — | Required when `AI_EMBEDDING_PROVIDER=openai`. |
| `AI_CHUNK_SIZE` | `512` | Target chunk size in tokens. |
| `AI_CHUNK_OVERLAP` | `50` | Overlap between consecutive chunks (tokens). |
| `AI_MAX_DOCUMENT_SIZE_MB` | `50` | Max file size for processing. |
| `AI_TOP_K_RESULTS` | `5` | Number of chunks returned for search. |

See `Backoffice/config/config.py` for the full AI/RAG section and any additional options.

## How vectors are used

- **Vector search:** The user query is embedded with the same embedding service; the app runs a similarity search (e.g. cosine) in pgvector and returns the top-k chunks.
- **Hybrid search:** Combines vector similarity and keyword search for better recall.
- **RAG:** Retrieved chunks are passed as context to the LLM when answering questions (chatbot, AI Document Library Q&A, workflow docs).

The AI Document Library page lets you choose **Vector only** or **Hybrid (keywords and embeddings)** for the "Ask" feature.
