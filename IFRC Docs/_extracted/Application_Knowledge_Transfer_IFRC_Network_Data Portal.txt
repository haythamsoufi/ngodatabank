2. Executive Summary
Purpose of the application: The IFRC Network Data Portal is a unified digital platform for managing federation-wide data collection processes. It provides a backoffice system for developing data collection forms, assigning forms to be filled by countries/National Societies, managing indicators, and providing integrated analytics and AI-assisted insights. It replaces multiple fragmented data collection tools (Kobo, Office Forms, Excel files, multiple one-off systems) with a single standardized, cost-effective solution.Business problem solved: The FDS team relied on scattered tools causing duplication, inconsistent standards, high maintenance costs, data loss, and limited accessibility. Different teams used different KPI names, definitions, and data structures. This system consolidates all federation-wide data collection processes (FDRS, Unified Planning, Unified Reporting, Fed-wide Emergencies) into one secure platform, enforcing the use of the Indicator Bank and making data available to the full IFRC network and the public.Key users / stakeholders:- IFRC Secretariat admins (FDS team, PMER teams, regional/country offices) - form design, assignment management, analytics- National Society focal points - data entry and submission- DT/IT teams - infrastructure, code review, security- Senior management - strategic reporting and decision-makingCurrent status: Phase 1 - Pre-launch. Code review and security validation underway. Infrastructure hosted on Azure App Services. Targeted rollout before the Unified Mid-Year Reporting data collection (April-May 2026).Known limitations:- Some advanced functionalities (AI chatbot, advanced integrations) will evolve after initial deployment- Full migration of historical legacy data is out of scope for initial launch

4. Application Architecture
Component breakdown (name, responsibility, dependencies):1. Flask Application (app/__init__.py) - Core web application factory. Creates the Flask app, registers blueprints, initializes extensions, configures middleware (CORS, CSRF, session management, security headers, API tracking). Depends on: Flask, SQLAlchemy, Flask-Login, all extensions.2. Authentication Module (app/routes/auth.py) - Local login (email/password with WTForms), Azure AD B2C OIDC/SSO, account lockout, password reset, rate limiting. Depends on: Flask-Login, Flask-WTF, PyJWT, Azure B2C tenant.3. Admin Interface (app/routes/admin/) - Modularized from a single monolithic file (340KB, 7000+ lines, 122 routes) into focused modules:   - Form Builder (form_builder/) - Template creation, section/item management, version control, KoBo import wizard (~40+ routes)   - User Management (user_management/) - User CRUD, role/entity assignments   - Assignment Management (assignment_management.py) - Form assignments to countries/entities, public form links, submission management (~25+ routes)   - Content Management (content_management.py) - Resources, publications, documents (~20+ routes)   - System Admin (system_admin/) - Countries, sectors, indicator bank, lookup lists (~30+ routes)   - Analytics (analytics.py, analytics_api.py) - Dashboard APIs, user activity tracking, system monitoring (~13+ routes)   - Utilities (utilities/) - Import/export, translations, sessions, CSRF handling (~20+ routes)   - Shared (shared.py) - Permission decorators (admin_required, permission_required), RBAC helpers4. Forms Module (app/routes/forms/) - Form entry pages, submission handling, Excel export, document attachments, matrix grid APIs, real-time presence tracking.5. API Module (app/routes/api/) - RESTful API v1 for external/website/mobile consumers: assignments, templates, data, submissions, documents, indicators, countries, users, mobile auth (JWT).6. AI Module (app/routes/ai.py, ai_ws.py, ai_documents/) - Chat API (HTTP + SSE + WebSocket), RAG document search and QA, AI agent executor, embedding service, reasoning traces, terminology service.7. Notification System (app/routes/notifications.py, notifications_ws.py) - User notifications via HTTP and WebSocket, push notifications via FCM, notification campaigns, email delivery.8. Services Layer (app/services/) - Business logic: form data processing, template management, AI chat request handling, AI vector store, AI embedding service, translation services (LibreTranslate, IFRC Translation API), email client (IFRC Email API), KoBo data import, authorization service, notification push (FCM).9. Models Layer (app/models/) - 50+ SQLAlchemy models across modules: core (User, Country), forms (FormTemplate, FormTemplateVersion, FormSection, FormItem, FormData), assignments (AssignedForm, AssignmentEntityStatus), indicator bank (IndicatorBank, Sector), organization hierarchy (NationalSociety, NSBranch), RBAC, AI models (AIConversation, AIDocument, AIEmbedding), system (Notification, SecurityEvent, AdminActionLog).10. Middleware (app/middleware/) - Session timeout, security headers, API tracking, transaction management.Data flow description:1. Admins design form templates using the Form Builder, adding indicators from the Indicator Bank, questions, and matrix tables with skip logic and validation rules.2. Templates are versioned and assigned to countries/entities for specific reporting periods.3. Focal points access assigned forms via the entry form interface, input data with real-time auto-save, presence detection, and validation.4. Submitted data is stored in PostgreSQL (FormData model with JSON payloads).5. Data is exposed via REST APIs to the public and authorized users for non-public data6. AI modules provide document intelligence (upload, embed, search, QA) and chat-based data exploration.7. Analytics dashboards aggregate data for reporting and decision-making.8. Notifications inform users of assignments, submissions, and system events via in-app, email, and push channels.

9. Security & Compliance
Authentication & authorization:- Authentication: Dual mechanism - Local login (email/password with WTForms, bcrypt password hashing) and Azure AD B2C OIDC/SSO (PKCE flow). Mobile app uses JWT bearer tokens (PyJWT).- Authorization: Role-based access control (RBAC) with database-backed permissions. Roles include admin, focal_point, view_only. Permission decorators (admin_required, permission_required) gate admin routes. Entity-level permissions (UserEntityPermission) control access per country/organization.- Session management: Flask sessions with HTTP-only secure cookies, SameSite policy. Automatic inactivity timeout (2 hours). Session blacklisting for security. Activity tracking per session.- Account security: Account lockout after repeated failed login attempts. Security events logged (SecurityEvent model). Rate limiting via Flask-Limiter on auth and password reset endpoints.Encryption:- Passwords: bcrypt hashing via Werkzeug security utilities- Transport: TLS 1.2+ enforced (Azure App Service)- Storage: Azure-managed encryption at rest for Blob Storage and PostgreSQL- Cookies: HTTP-only, Secure flag, SameSite policy- CSRF tokens: Double-submit cookie pattern via Flask-WTF CSRFProtectKnown risks:- B2C authentication readiness and configuration complexity- API vulnerabilities requiring ongoing penetration testing- Dependency on external AI provider API keys for AI features- Public form submissions (no auth) require careful validationCompliance considerations:- CSRF protection enabled globally with per-route enforcement patterns- Security headers middleware (CSP with nonces for inline scripts, X-Frame-Options, etc.)- CORS restricted to configured origins (no wildcard in production)- Content Security Policy with nonce-based inline script allowance- Rate limiting on sensitive endpoints (login, password reset, API)- Comprehensive audit logging (AdminActionLog, UserActivityLog, SecurityEvent)- Input sanitization via bleach library- CodeQL security analysis and dependency auditing in CI/CD- Penetration testing planned (March 2026)
10. Deployment & Release Management
CI/CD pipeline:- GitHub Actions workflows:  - deploy-to-webapp.yml: Manual deployment to Azure App Service (staging or production). Validates publish profile, builds Docker image, pushes to Azure Container Registry (ifrcimage.azurecr.io/databank_backend:<version>), deploys to target App Service.  - test.yml: On push/PR to main/develop. Python 3.11, PostgreSQL 14 service container, pytest with coverage, Codecov reporting, 50% coverage gate.  - codeql.yml: CodeQL security analysis for JavaScript/TypeScript and Python on PRs to main.  - security-scan.yml: Gitleaks secret scanning, Bandit Python security analysis.  - dependency-audit-on-demand.yml: Manual pip-audit and npm audit.  - dependency-review-on-pr.yml: GitHub Dependency Review on PRs.  - load-database-dump-to-stage.yml: Manual database restore to staging environment.Deployment steps:1. Developer pushes to feature branch and creates PR to develop/main2. CI runs tests, CodeQL, and dependency checks automatically3. After merge, authorized user triggers deploy-to-webapp.yml workflow manually4. Workflow selects environment (Staging or Production) and version tag5. Docker image is built with multi-stage Dockerfile (Python 3.11-slim base)6. Image is pushed to Azure Container Registry (ifrcimage.azurecr.io)7. Azure Web App is updated with new container image8. Container entrypoint.sh runs: waits for DB, runs migrations (flask db upgrade), optional RBAC seed, starts GunicornRollback procedure:- Re-deploy previous Docker image version from Azure Container Registry- If migration issues: set MIGRATION_ALLOW_ROLLBACK env var and run flask db downgrade- Container entrypoint supports SKIP_MIGRATIONS and WAIT_FOR_MIGRATIONS for controlled rollouts

Application Insights integration available.- Client-side: Controlled console logging via CLIENT_CONSOLE_LOGGING flag and console guard system.- Health check: /health endpoint for container health monitoring.Common issues & troubleshooting:- CSS changes not appearing: Run 'npm run watch:css' or 'npm run build:css' in Backoffice/ to regenerate TailwindCSS output.css.- Flask commands fail with FLASK_APP error (Windows/PowerShell): Set  = 'run.py'- AI features return 'no provider configured': Ensure at least one AI provider key is set (OPENAI_API_KEY, GEMINI_API_KEY, or Azure equivalents).- RAG returns nothing after model change: AI_EMBEDDING_DIMENSIONS must match pgvector column; changing requires migration + re-embedding.- WebSocket not working: Ensure flask-sock is installed; HTTP/SSE endpoints still work without it.- Multiple Alembic heads: Run 'flask db heads' before any migration work; resolve branch points before proceeding.

13. Performance & Scalability
Bottlenecks:- Large form rendering with many sections/items and complex skip logic can be computationally intensive- AI embedding generation and vector similarity search (pgvector) on large document corpora- Excel import/export for large datasets (handled by openpyxl/XlsxWriter with streaming where possible)- Translation API calls for bulk auto-translate operations across 7 languagesScaling approach:- Gunicorn with gthread workers (configurable via GUNICORN_WORKERS, GUNICORN_THREADS)- PostgreSQL connection pooling (SQLALCHEMY_POOL_SIZE, SQLALCHEMY_MAX_OVERFLOW, SQLALCHEMY_POOL_TIMEOUT)- Redis caching (optional) for rate limiting, presence tracking, and cross-worker coordination- DiskCache for local caching when Redis is unavailable- Azure App Service scaling (vertical and horizontal via Azure configuration)- Static file serving with asset versioning for browser caching- Flask-Compress for response compression in non-development environmentsPerformance tuning:- Database query optimization with SQLAlchemy relationship loading strategies- Translation caching to reduce API calls- Form state management optimization (pagination state restoration for large forms)- AJAX auto-saving in entry forms to prevent data loss- Presence tracking uses cache/memory (not database) to avoid write-heavy operations- APScheduler for background tasks (session cleanup, scheduled operations)
14. Known Limitations & Technical Debt
Short-term workarounds:- Excel export/import features are partially developed; some edge cases require manual data handling- Some AI features depend on specific model availability (e.g., GPT-5 rejects sampling params)- Historical data from legacy systems (FDRS, KoBo) requires manual migration scriptsLong-term recommendations:- Implement comprehensive end-to-end browser testing (Playwright) in CI pipeline- Add frontend JavaScript unit testing framework- Increase test coverage above 50% threshold- Plan and execute historical data migration from legacy FDRS system- Evaluate dedicated search infrastructure (Elasticsearch) if RAG corpus grows significantly- Consider horizontal scaling architecture (multiple Gunicorn instances behind load balancer) for production growth- Establish formal data retention policies- Complete penetration testing and address all findings before production launch
